Typed workspace dependency DAG with verification proofs for cross-workspace
infrastructure dependencies.

## Purpose

The pleme-io ecosystem has 14+ Pangea workspaces (seph-vpc, seph-cluster,
pleme-dns, etc.) that reference each other via Terraform remote state. This
crate provides typed Rust representations of workspace inputs/outputs and
proves graph-level invariants: connectivity, ordering, type compatibility,
and uniqueness.

## Crate Structure

## Key Types and Relationships

### Core Graph Types (`types.rs`)

- `WorkspaceId(String)` -- Content-addressable identity. Ord + Hash + Serialize.
- `OutputPort { name, field_type: IacType, source_resource }` -- What a workspace produces. `source_resource` is the Terraform resource path (e.g., `aws_route53_zone.zone.zone_id`).
- `InputPort { name, field_type: IacType, source_workspace: WorkspaceId, source_output }` -- What a workspace consumes. Points to a specific output on a specific workspace.
- `WorkspaceNode { id, name, state_key, provider, outputs, inputs }` -- A workspace with its ports. `state_key` is the S3 state file key. `provider` is `aws`, `akeyless`, etc.
- `WorkspaceEdge { from, to, output_name, input_name, field_type }` -- A typed dependency between workspaces. Auto-generated by the builder from input declarations.
- `WorkspaceGraph { nodes: BTreeMap<WorkspaceId, WorkspaceNode>, edges: Vec<WorkspaceEdge> }` -- The complete graph. BTreeMap ensures deterministic iteration order.

All types implement `Serialize`/`Deserialize`. Port types use `iac_forge::ir::IacType` (the shared type system across all IaC forge backends).

### Builder (`builder.rs`)

`WorkspaceGraphBuilder` -- fluent builder. Accumulates workspaces, pending outputs, and pending inputs. On `.build()`:
1. Attaches pending outputs to their workspace nodes
2. Attaches pending inputs to their workspace nodes
3. Auto-generates `WorkspaceEdge` for each input declaration

Methods: `.workspace(id, name, state_key, provider)`, `.output(workspace, name, type, source)`, `.input(workspace, name, type, source_ws, source_output)`, `.build()`.

### Composition Model (`composition.rs`)

Maps the convergence process model to infrastructure state boundaries:

- `CompositionBuilder` -- Fluent builder for decomposing architectures into sub-workspaces.
  - `::new(name, label, provider)` -- PID 1 is the parent (orchestrator).
  - `.sub_workspace(name, |ws| ...)` -- Add a child process. PIDs assigned sequentially from 2.
  - `.build()` -- Produces a `CompositionPlan`.

- `SubWorkspaceBuilder` -- Configures ports for a single sub-workspace.
  - `.output(name, type, source)` -- Declare an output.
  - `.input_from(source_ws, output_name, type)` -- Input from sibling (same name).
  - `.input_as(local_name, source_ws, source_output, type)` -- Input with different local name.

- `CompositionPlan` -- The resolved composition.
  - `.verify()` -- Runs `verify_graph` on the internal graph.
  - `.deployment_order()` -- Topological sort.
  - `.process(pid)` -- Lookup child by PID.
  - `.process_by_name(name)` -- Lookup child by name.
  - Fields: `parent: WorkspaceId`, `children: Vec<ChildProcess>`, `graph: WorkspaceGraph`.

- `ChildProcess` -- A child workspace in the composition.
  - Fields: `id: WorkspaceId`, `name`, `pid: u32`, `ppid: u32`, `state_key`.
  - `ppid` is always 1 (parent). PIDs start at 2.

**Name resolution:** Sub-workspace IDs are `{parent}-{child}` (e.g., `seph-network`). When a sub-workspace references a sibling via `input_from("network", ...)`, the builder resolves this to `seph-network`. State keys are `pangea/{parent}/{child}`.

### Process Model Analogy

| Process Model | Composition |
|---------------|-------------|
| PID 1 (init) | Parent workspace (orchestrator, no resources) |
| PID 2+ | Child workspaces (own Terraform state each) |
| PPID | Always 1 (flat tree within a composition) |
| IPC | Typed ports (OutputPort -> InputPort) |
| fork() | `.sub_workspace()` call |
| Process table | `CompositionPlan.children` |
| Process start order | `deployment_order()` |

### Verification (`verify.rs`)

`GraphViolation` enum variants:
- `DisconnectedInput` -- Input references a nonexistent workspace or output.
- `TypeMismatch` -- Output type differs from input type on a connected edge.
- `CyclicDependency` -- Graph contains a cycle (Kahn's algorithm).
- `OrphanWorkspace` -- Workspace with no inputs or outputs (defined but unused).
- `DuplicateOutput` -- Same output name appears twice in one workspace.

Functions:
- `verify_graph(graph)` -- Runs all four checks, returns `Ok(())` or `Err(Vec<GraphViolation>)`.
- `verify_connectivity(graph)` -- Every input has a matching output in its source workspace.
- `verify_compatibility(graph)` -- Output IacType matches input IacType for each edge.
- `verify_ordering(graph)` -- No cycles (topological sort must exist).
- `verify_uniqueness(graph)` -- No duplicate output names within a workspace.

### Analysis (`analysis.rs`)

- `deployment_order(graph)` -- Kahn's algorithm topological sort. Returns `Result<Vec<WorkspaceId>, GraphViolation>`. Error if cycle detected.
- `affected_workspaces(graph, changed)` -- BFS transitive closure of all dependents. Changed workspace is not included in the result.

## Real pleme-io Topology (`pleme.rs`)

`pleme_infrastructure_graph()` returns the actual 6-workspace dependency graph:

```
state-backend          -- S3 bucket + DynamoDB lock table (no deps)
pleme-dns              -- Route53 zone (independent)
nix-builders           -- Nix builder fleet (depends on pleme-dns)
seph-vpc               -- VPC, subnets, security group (independent)
seph-cluster           -- K3s cluster (depends on seph-vpc + pleme-dns)
akeyless-dev-config    -- Akeyless gateway (depends on seph-cluster)
```

Edges (5 total):
- pleme-dns.zone_id -> nix-builders.zone_id (String)
- seph-vpc.vpc_id -> seph-cluster.vpc_id (String)
- seph-vpc.private_subnet_ids -> seph-cluster.subnet_ids (List(String))
- pleme-dns.zone_id -> seph-cluster.dns_zone_id (String)
- seph-cluster.cluster_endpoint -> akeyless-dev-config.cluster_endpoint (String)

`builder_fleet_composition()` decomposes nix-builders into 4 sub-workspaces:
- network -- VPC, subnets
- security -- security group, IAM instance profile (depends on network)
- compute -- NLB, ASG (depends on network + security)
- dns -- Route53 CNAME (depends on compute)

`minimal_graph()` returns a 3-workspace test fixture (vpc + dns + cluster).

## Integration Points

### iac-forge (`IacType`)

All port types use `iac_forge::ir::IacType` -- the canonical type enum shared across
terraform-forge, pulumi-forge, crossplane-forge, ansible-forge, pangea-forge, and
steampipe-forge. This ensures type compatibility verification uses the same type
system as actual IaC code generation.

### ruby-synthesizer / pangea-sim

The workspace graph can drive simulation: each node's outputs map to synthesized
Terraform resources, and verification proves the wiring is correct before any
real infrastructure is deployed.

### convergence-controller

`CompositionPlan` maps directly to the convergence process model. Child PIDs
align with the `ConvergenceProcess` CRD's PID field. State keys map to S3
state file paths managed by pangea-operator.

## Testing

111 tests total. Run: `cargo test`

Property tests (proptest) use two strategies:
- `arb_linear_graph` / `arb_linear_composition` -- Linear chain (w0 -> w1 -> ... -> wN)
- `arb_fanout_graph` / `arb_fanout_composition` -- One root, multiple leaves

The 10 numbered composition proofs verify:
1. Valid compositions always pass verification
2. Removing an output breaks connectivity
3. Type mismatches are detected across compositions
4. Deployment order is a valid topological sort
5. Impact analysis is transitively complete
6. PIDs are unique and sequential
7. State keys follow `pangea/{parent}/{child}`
8. Empty composition is valid
9. Diamond dependencies are handled
10. Self-referencing compositions are rejected
